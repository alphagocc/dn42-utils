## Findings

### [P2] Require Pydantic v2 when importing `field_validator`

`src/dn42ctl/api.py` now imports `field_validator`, which is only available in Pydantic v2, but `pyproject.toml` still only depends on `fastapi>=0.115.0`. FastAPI versions in that range can be installed with Pydantic v1, so environments that already satisfy FastAPI with Pydantic 1.x will fail importing `dn42ctl.api` and `dn42ctl serve` will not start. Add an explicit `pydantic>=2` dependency or use v1-compatible validators.

Location: `src/dn42ctl/api.py:7`

### [P2] Preserve normalized `babel_type` in service calls

`validate_babel_type()` lowercases valid values, but `create_ibgp_peer()` and `modify_ibgp_peer()` call it without assigning the returned value. When the exported service layer is called directly with `babel_type="TUNNEL"`, validation now succeeds but the uppercase value is persisted and rendered as `type TUNNEL;`, instead of the normalized `tunnel`. Assign the validator result before inserting/updating and rendering.

Locations: `src/dn42ctl/services/ibgp.py:91-92`, `src/dn42ctl/services/ibgp.py:250-251`

### [P2] Validate CLI BGP peer ASN before creating files

The new validation layer documents ASN as a positive integer and validates API/config ASN values, but `dn42ctl bgp peer --asn ...` still does not call `validate_asn`. A CLI value such as `--asn -1` can still reach `create_bgp_peer()`, derive `ifname=dn42_-1`, and render an invalid Bird peer AS. Add `validate_asn(peer_asn)` in the CLI create path before calling the service.

Location: `src/dn42ctl/cli.py:595-598`

## Overall

The patch improves validation coverage, but it is not fully correct as-is because the API now relies on an undeclared Pydantic v2 dependency, service-level normalization is dropped for direct callers, and the CLI BGP create path still misses ASN validation.
