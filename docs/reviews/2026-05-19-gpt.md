## Findings

### [P1] Node proposal/report import paths bypass service-layer validation

Node-submitted payloads can reach peer creation with little validation. In
`proposal_decisions.py`, raw JSON values are coerced through `str()` / `int()`
and passed into `create_bgp_peer()` / `create_ibgp_peer()`. `report_import.py`
does the same for imported `scan_result` reports. API and CLI validators protect
direct admin calls, but they do not protect node proposals, auto-accepted
proposals, or imported reports.

This matters because those values are later rendered into templates such as
`networkd_netdev.j2` and Bird peer configs. A node token with
`write_policy.peer_add=auto_accept`, or an accepted/imported malicious payload,
can put invalid or newline-bearing values into generated config.

Locations:

- `src/dn42ctl/services/proposal_decisions.py:61`
- `src/dn42ctl/services/report_import.py:108`
- `src/dn42ctl/templates/networkd_netdev.j2:11`
- `src/dn42ctl/templates/bird_bgp_peer.conf.j2:6`

Recommendation: move input validation/normalization into the service layer
called by all mutation paths, or validate proposal/report payloads before they
are persisted or applied.

### [P1] Peer create/modify/delete is not atomic across DB and filesystem

The peer mutation services update SQLite and write/delete config files in
separate steps with no rollback strategy. For example, `create_bgp_peer()`
inserts the DB row before rendering files, `modify_bgp_peer()` updates the DB
before file writes, and `delete_bgp_peer()` deletes files before deleting the DB
row. iBGP has the same pattern.

If a permission error, template error, or partial filesystem failure occurs, the
database and `/etc` can diverge. That is especially risky for a tool whose
SQLite state is the authority for later `show`, `genconf`, `desired`, and node
apply workflows.

Locations:

- `src/dn42ctl/services/bgp.py:82`
- `src/dn42ctl/services/bgp.py:176`
- `src/dn42ctl/services/bgp.py:239`
- `src/dn42ctl/services/ibgp.py:112`
- `src/dn42ctl/services/ibgp.py:216`
- `src/dn42ctl/services/ibgp.py:300`

Recommendation: stage file content first, perform atomic writes, and only commit
DB changes once filesystem operations succeed; or wrap DB changes in an explicit
transaction and compensate on render/write failures.

### [P2] Desired-state revision is not content-stable

`_compute_revision()` appends `generated_at` to the content hash. As a result,
unchanged desired-state content pulled in a later second creates a different
revision and a new history row. Existing tests only compare the hash suffix, so
this behavior is not caught.

Location:

- `src/dn42ctl/services/desired_state.py:96`

Recommendation: use a content-derived revision id for identity, and store
`generated_at` separately as metadata. If a timestamp prefix is desired for
display, avoid using it in uniqueness or sync equality semantics.

### [P2] `node remove` does not cascade as documented

`ManagedNodeStore.delete()` deletes only from `managed_nodes`. The related
`config_proposals`, `node_reports`, and `config_revisions` tables reference the
parent `nodes` table, not `managed_nodes`, so those rows remain after
`dn42ctl node remove`.

This contradicts the command documentation, which says removal cascades through
proposals, reports, and revisions. Tests only verify cascade when deleting from
`nodes` directly, not through the actual `remove_node()` service.

Locations:

- `src/dn42ctl/db_managed.py:242`
- `src/dn42ctl/migrations.py:99`
- `docs/commands/node.md:32`
- `tests/test_db_managed.py:223`

Recommendation: decide whether `node remove` should preserve history or truly
delete the parent `nodes` row, then align service behavior, docs, and tests.

### [P2] SQLite connections are leaked in multiple service paths

The full test suite passes but emits many `ResourceWarning: unclosed database`
warnings. Targeted tracemalloc runs confirmed allocations from service functions
that call `open_db()` / `open_db_and_ensure_node()` and never close the returned
connection.

Examples:

- `src/dn42ctl/services/bgp.py:54`
- `src/dn42ctl/services/show.py:94`
- `src/dn42ctl/services/init_sys.py:64`
- `src/dn42ctl/services/scan.py:407`

Recommendation: make `Database` a context manager or consistently wrap service
usage in `try/finally: db.close()`. Add a warning-as-error test for at least one
representative API/service path.

### [P2] `validate_ipv6_address()` accepts CIDR strings in address-only fields

`validate_ipv6_address()` validates only the part before `/` and then returns
the original value unchanged. This is intentional in one existing test, but the
same validator is used for fields like peer LLA, which is rendered into Bird
neighbor syntax. A value such as `fe80::1/64` is accepted but would produce an
invalid neighbor line.

Locations:

- `src/dn42ctl/validators.py:64`
- `tests/test_validators.py:118`
- `src/dn42ctl/templates/bird_bgp_peer.conf.j2:6`

Recommendation: split address and interface/CIDR validation into separate
helpers, then use the address-only helper for peer LLA and peer IP fields.

### [P3] Node apply override key naming is inconsistent

`node_apply._resolve_paths()` says `bird_peers_dir` is recognized, but the code
only reads `peers_dir`. A user following the docstring or the `AppConfig` field
name gets a silent no-op for that override.

Locations:

- `src/dn42ctl/services/node_apply.py:64`
- `src/dn42ctl/services/node_apply.py:77`
- `tests/test_node_config.py:40`

Recommendation: accept both `peers_dir` and `bird_peers_dir`, or document one
canonical key and reject unknown `[apply]` keys instead of silently ignoring
them.

## Notes

The static checks are clean and the Python test suite is broad, but current
coverage does not exercise some important failure modes:

- filesystem write failure after DB mutation,
- proposal/report payload injection,
- warning-as-error checks for leaked SQLite connections,
- `remove_node()` cascade behavior,
- exact desired-state revision stability.
