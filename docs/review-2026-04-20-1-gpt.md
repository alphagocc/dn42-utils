## MUST：规格符合性确认（Spec Compliance）

对照 `docs/spec.md` 的 MUST 条款与命令规范，当前实现整体满足“功能覆盖 + 关键约束”要求，且计划中关键重构点均已落地。

- 功能覆盖（CLI 命令集）
  - 已具备：`init`、`genconf`、`scan`、`bgp peer` / `bgp peer modify`、`ibgp peer` / `ibgp peer modify`、`show`（wg/bgp/ibgp/all）、`del peer`（bgp/ibgp）。
  - CLI smoke check：`uv run dn42ctl --help` / `--help` 子命令均可正常加载。
- 网络后端与强制约束
  - networkd：模板中固定包含 `RouteTable=off`。
  - NetworkManager：模板中固定包含 `peer-routes=false`。
  - AllowedIPs：networkd/NM 模板均会写入（默认 `fe80::/64` 与 `fd00::/8`，并可承载 DB 中的列表）。
- `scan` 范围与行为
  - 仅扫描/导入 networkd 与 NetworkManager；未发现对 `/etc/wireguard`（wg-quick）的扫描逻辑。
  - 支持 best-effort：大量文件中单个文件解析失败不会中断全局（通过 `skipped`/`warnings` 聚合）。
- Render 层
  - 已使用 Jinja2（`pyproject.toml` 含 `jinja2>=3.0.0`，`render.py` 使用 `PackageLoader` 加载模板）。
- Babel `rxcost` 设计
  - DB：迁移 v2 为 `ibgp_peers` 增加 `babel_rxcost`，默认 120。
  - CLI：`ibgp peer` 必填/交互提示 `--rxcost`；`ibgp peer modify` 支持更新并重生成 `babel.conf`。
  - Render：`babel.conf.j2` 按 (ifname, rxcost) 渲染每个 interface。
  - Scan：会尽力从 `babel.conf` 解析各接口 `rxcost` 并导入 DB，缺失则 warning + 回退默认值。

已发现的“接近规格但存在偏差/风险”的点：

- `bgp peer modify` 在 CLI 侧允许把 `peer_lla` 清空（空字符串）并绕过校验；而渲染层要求 `peer_lla` 非空。这会触发错误路径，并可能造成 DB 与文件状态不一致（详见下文「逻辑与边界」）。
- `scan` 的部分错误信息未包含“文件路径”，与 `docs/commands/scan.md` 的输出约定不完全一致（详见下文「逻辑与边界」）。

## 1. 逻辑与边界 (Logic & Edge Cases)

### 1.1 P0：可导致状态不一致的修改路径（BGP peer modify）

现象：`cmd_bgp_peer_modify` 中对 `peer_lla` 的校验逻辑是：仅当 `peer_lla` 为 truthy 时才调用 `_validate_peer_lla()`；因此用户若在交互提示中输入空字符串，校验会被绕过。

- 触发链路：
  - `src/dn42ctl/cli.py`：
    - `peer_lla = typer.prompt("Peer LLA", default=str(row["peer_lla"] or ""))`
    - `if peer_lla: peer_lla = _validate_peer_lla(peer_lla)`
  - `src/dn42ctl/services/bgp.py`：
    - `modify_bgp_peer()` 先 `db.update_bgp_peer(... peer_lla=peer_lla ...)`
    - 再调用 `render_bird_bgp_peer_conf(peer_lla=peer_lla, ...)`，而 `render_bird_bgp_peer_conf` 对空值 `peer_lla` 会抛 `ValueError`。

风险：

- DB 更新先于渲染写文件：当 `peer_lla` 为空时，渲染失败发生在 DB 已更新之后，导致 DB 中 `peer_lla` 变为空字符串，但文件未重生成/未同步，形成“半状态”。
- 后续影响：再次 modify / show / genconf 的行为可能变得不可预测（例如一直因 `peer_lla` 空而无法渲染）。

建议（按侵入性从低到高）：

1. CLI：对 `bgp peer modify` 强制要求 `peer_lla` 非空（不允许清空）。
2. Service：在写 DB 之前做同样的非空校验，避免被未来 REST/脚本调用绕过。
3. 原子性：更彻底的做法是“先渲染并写入临时文件、成功后再提交 DB”，或失败回滚 DB（见「一致性」相关建议）。

### 1.2 Scan：错误恢复与信息粒度

整体上 `scan_local_configs()` 已遵循 best-effort：文件缺失/读失败/解析失败会进入 `skipped` 或 `warnings`，不会中断扫描。

但与 `docs/commands/scan.md` 的“默认输出文件路径 + 简短错误信息”约定相比，仍有细节偏差：

- NM 配置解析失败（`configparser.Error`）时，当前记录为：`"{ifname}: 解析 NM 配置失败: {exc}"`，缺少具体 `nm_path`。
- 多数 “缺少字段” 类 skipped 只包含 `ifname`，不包含源文件路径；这对于排障会更困难（尤其当目录中存在多个同名/覆盖关系的配置文件时）。

建议：

- 对“读/解析失败”这类属于坏文件的情况，记录具体文件路径（`netdev_path/network_path/nm_path/bird_path/babel_path`）。
- 对“字段缺失”类 skipped，可以保持 `ifname` 粒度即可（不必过度冗长），但至少在解析失败时携带路径。

### 1.3 其它边界点（值得记录）

- `create_bgp_peer()` 创建时不会检查 ListenPort 与现存 peers 冲突（`modify_bgp_peer()` 会检查）。
  - 若用户已有 iBGP peer 占用同端口，可能产生运行态冲突（非 DB 层冲突）。
  - 若这是“按 ASN 推导端口”的刻意设计，可在文档中说明“端口冲突由用户自行处理”；否则建议在创建时做一次 best-effort 检查。
- `modify_bgp_peer()` 直接 `json.loads(row["allowed_ips_json"])`，若 DB 被手工修改/损坏导致 JSON 不合法，会抛出未捕获异常。
  - 作为运维工具，遇到坏 DB 更应转换为可理解的错误（或回退默认值并 warning），而不是直接抛 traceback。

## 2. 性能与复杂度 (Performance & Efficiency)

### 2.1 `show all` 存在重复探测与重复查询

当前 `cmd_show_all` 同时调用：

- `show_wg_tunnels()`（其内部又调用 `show_bgp_peers()` + `show_ibgp_peers()`）
- `show_bgp_peers()`
- `show_ibgp_peers()`

这会造成：

- DB 查询重复（至少两轮 list）。
- live probes 重复：`wg show` / `birdc show protocols` 会被执行多次。

在 peers 数量较多时，重复的外部命令调用会放大耗时，并增加系统负载。

建议：

- 在 `show all` 中只调用一次 `show_bgp_peers/show_ibgp_peers`，然后在 CLI 层拼装 `wg` 视图；或提供一个“单次探测，返回 wg/bgp/ibgp 三套视图”的 service API。

### 2.2 `show wg` 不必要触发 `birdc`

`show_wg_tunnels()` 为了复用结构，复用 `show_bgp_peers/show_ibgp_peers(include_live=True)`。

- 但 `WgTunnelView` 不包含 Bird live 信息，导致 `birdc` 探测结果被计算后丢弃。

建议：

- 为 live probe 增加更细粒度开关（只跑 wg 或只跑 bird），或为 `show wg` 写一个只调用 `wg show` 的路径。

### 2.3 复杂度总体评价

- 绝大多数操作是本地小文件 I/O + SQLite；复杂度为 O(N)（N 为 peers 数），合理。
- `scan` 通过候选集合减少无谓遍历；对大量文件时也属于线性复杂度。
- `show` 已引入 `ThreadPoolExecutor` 并为外部命令设置 timeout，符合 plan 中“并发探测”的优化目标。

## 3. 安全与健壮性 (Security & Robustness)

### 3.1 外部输入校验：总体有，但仍有“配置注入”窗口

正向点：

- `pubkey` 使用 base64 形态校验（长度与字符集），能阻断大多数明显非法输入。
- `peer_lla` 使用 `ipaddress.IPv6Address` 校验，能阻断非法 IPv6。
- service 层对 `listen_port`、`rxcost` 有范围校验（0/1-65535）。

风险点：

- `endpoint` 校验允许空（可能是设计允许“无 Endpoint”）；但当前正则对 host 部分限制较弱，可能容许空白字符（包括换行）进入。
  - 一旦 endpoint 中包含换行，渲染到 networkd/NM 配置文件可能导致“额外键值”被注入（配置注入风险），或导致配置解析异常。

建议：

- endpoint 至少应禁止任何空白字符（`\s`），并明确支持两种形态：`host:port` 与 `[IPv6]:port`。
- 同时建议在 service 层再次校验（为未来 REST 接口复用做准备），避免绕过 CLI 校验。

### 3.2 外部命令 timeout：show 已做，key/scan 仍可补齐

现状：

- `show` 的 `wg show` / `birdc` 已设置短 timeout（2s），且失败不会让 show 命令整体失败。

待加强点：

- `wg genkey` / `wg pubkey`（生成/推导公钥）未设置 timeout：
  - `src/dn42ctl/wg.py:generate_wg_keypair`
  - `src/dn42ctl/services/scan.py:_wg_pubkey_from_private`

理论上外部命令挂起会导致 CLI 永久阻塞（尤其在异常 PATH、卡死的 wg 工具、或系统资源极差时）。

建议：

- 对 key 相关的 `subprocess.check_output()` 也设置短 timeout（例如 2~5s），并把超时转成可理解的 `Dn42CtlError/WireGuardError`。

### 3.3 文件写入的原子性与一致性

目前 `write_text()` 直接覆盖写目标文件。

- 当磁盘写满、进程被 kill、或目标目录存在 I/O 问题时，可能产生“部分写入”的截断文件。
- 结合 service 中“先写 DB 再写文件”的流程，会出现 DB/文件不一致的风险。

建议：

- 使用临时文件写入 + 原子 `replace()`（POSIX 原子 rename）替换目标文件。
- 或在流程上选择一个 source-of-truth（DB 或文件），并对失败路径做恢复策略（记录失败、回滚、或延迟重渲染）。

### 3.4 systemd unit 文本拼接的健壮性

`genconf` 会在 Linux + systemd 可用时生成 ROA 更新 unit。当前 unit 中对路径做了引号处理，但路径来源可由用户通过 config 覆盖。

风险：

- 若路径包含引号/特殊字符，unit 解析可能失败；甚至可能意外改变参数分词。

建议：

- 明确限制这些路径不得包含换行与控制字符；必要时对 systemd ExecStart 参数做更严格的 escaping（或直接约束为“无空格路径”）。

## 4. 极简主义与冗余 (Minimalism & Redundancy)

### 4.1 过度设计与样板代码

- Service 层 view dataclass（`BgpPeerView/IbgpPeerView/WgTunnelView`）与 DB record 字段存在一定重复。
  - 从“未来 REST API 复用/脱敏/附加 live 状态”的角度看是合理分层；
  - 但同时带来样板代码与字段搬运成本。

建议：

- 维持现状即可；当确实要接 REST 层时再决定是否引入更统一的 model/mapper。

### 4.2 可能的死代码/遗留分支：`wgquick`

- `peer_files_for_backend()` 仍保留 `net_backend == "wgquick"` 的分支，并引用 `/etc/wireguard`。
- 当前 spec 与命令文档明确：scan 不支持 wg-quick；create/modify 也只允许 networkd/nm。

这段逻辑在“新创建数据”路径上几乎不可达，更像兼容历史 DB 的遗留。

建议（二选一）：

- 若明确不需要向后兼容：移除该分支，避免误导与维护成本。
- 若需要兼容旧 DB：保留，但建议在文档/代码中标注这是“仅用于历史记录展示/删除”的兼容路径，并确保不会被新流程写入。

### 4.3 仓库层面的冗余（非核心，但会影响使用者心智负担）

- 根目录存在多个 `*_template`、脚本（如 `create_peer.py`、`setup_dn42_roa.sh`）。
  - 若它们与当前 spec/实现不一致（例如缺失强制约束），可能误导用户。

建议：

- 将其标记为 legacy（移动到 `docs/` 或 `examples/` 并写明不再维护），或删除以减少误用面。

## 5. 标准与作用域 (Standards & Scope)

### 5.1 Python 语言规范与工程标准

- 使用 `from __future__ import annotations`、类型标注、`dataclass(frozen=True)`（在视图/记录类上）整体良好。
- DB 操作使用参数化查询，规避 SQL 注入。
- 错误类型分层清晰：CLI 处理 `typer.BadParameter` 与 `Dn42CtlError`；DB 层抛 `DatabaseError`。

### 5.2 作用域与突变控制

- 配置对象 `AppConfig` 是冻结 dataclass；CLI 更新 paths 使用 `dataclasses.replace()`，避免原地突变。
- Service 函数内部的临时变量作用域较小，整体符合“最小作用域”原则。

### 5.3 值得统一的小风格点

- `show.py` 的 `ThreadPoolExecutor` 从 `concurrent.futures.thread` 导入而不是从 `concurrent.futures` 导入，功能等价但不常见。
- 部分模块中 `dict[str, object]` + `cast()` 的写法可以接受（用于 best-effort parser），但若未来要强化类型检查，可以考虑引入更明确的 TypedDict/小 dataclass。
