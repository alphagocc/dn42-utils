## 2. 架构分层与 REST 复用（API Layering）

**优点**

- `services.py` 提供结构化返回（`InitResult/PeerResult/*View/DeleteResult/ScanResult`），CLI 可直接 `asdict()` 输出 JSON，符合 “未来 RESTful API 复用” 的方向。
- `render.py` 基本是纯函数渲染，利于单元测试与幂等性验证。

**可改进点（影响长期可维护性）**

- CLI 在 `bgp peer modify` 里**直接打开 DB 并读取记录**用于 prompt 默认值（`AppContext.open_db()` + `db.get_bgp_peer`），与“CLI 只做交互、service 做业务”的边界略不一致。
  - 建议：新增 `services.get_bgp_peer_view/get_bgp_peer_record`（或一个 `services.load_bgp_peer_for_modify`）让 CLI 不直接依赖 `db.py`。
- service 层当前每个函数内部都 `_open_db()`，`show all` 会多次打开 DB（`show_wg_tunnels` 又会调用 `show_bgp_peers/show_ibgp_peers`）。
  - 建议：中期可考虑在 service 层增加可选 `db: Database` 参数，或在 `AppContext` 里统一生命周期，避免重复连接/迁移。

## 3. 一致性与原子性（DB ↔ Files ↔ systemd）

这是当前最需要“严格审视”的部分，因为它直接决定生产环境可用性。

### 3.1 典型两阶段风险

- `create_bgp_peer/create_ibgp_peer`：**先 insert DB，再写 Bird/Network config 文件**。
  - 风险：写文件阶段若因权限不足、磁盘满、路径不可用而失败，会留下“DB 已存在但文件未生成”的不一致状态。
- `modify_bgp_peer`：**先 update DB，再覆盖写文件**。
  - 风险：文件写失败后 DB 已更新，文件仍是旧配置（或部分覆盖），造成“DB 与真实配置不一致”。
- `delete_bgp_peer/delete_ibgp_peer`：**先删除文件，再删除 DB**。
  - 风险：若 DB 删除失败（I/O/锁/损坏），可能出现“文件已删但 DB 仍记录存在”。

### 3.2 建议的改进方向（不改 UX，只增强可靠性）

- **确定一个“source of truth”并贯彻**：
  - 方案 A（偏运维）：以 DB 为真。所有命令先写 DB，文件写入失败则记录“待修复/生成失败”状态（例如 DB 增加 `last_render_error` / `rendered_at` 字段），并提供“重渲染/修复”能力。
  - 方案 B（偏保守）：以文件落地为真。先生成并落地文件（写到临时文件再原子 rename），成功后再提交 DB。
- 至少做到：
  - 文件写入使用 “写临时文件 → `replace()` 原子替换” 避免半写入；
  - 删除操作尽量 best-effort：聚合删除失败列表，最后再处理 DB 变更（或反过来，但要明确策略）。

> 说明：当前实现不一定“错误”，但在遇到权限/异常时会留下不可预测的部分状态；这在“接管既有环境（scan）”场景尤其常见。

## 4. 输入校验与交互 UX（Validation & UX）

**现状**

- `peer_lla` 在渲染 Bird peer conf 时做了非空校验（空会抛错），能避免生成致命的 Bird 语法错误（👍）。
- `endpoint` 在 networkd/NM 渲染时做了“空则省略”处理，避免 `Endpoint=` 空值导致配置无效（👍）。

**风险点**

- `peer_public_key`、`endpoint`、`peer_lla` **缺少格式校验**：
  - 空字符串/格式错误会被写入 DB，并可能写入 networkd/NM 配置（其中 PublicKey 为空基本必坏）。
  - 建议：最小校验即可（不做复杂解析）：
    - `peer_public_key`：非空、base64 字符集、长度范围（WG key 常见 44 chars，但不必写死）；
    - `endpoint`：允许空；非空时要求包含 `host:port` 且 `port` 为 1~65535；
    - `peer_lla`：非空时要求看起来像 IPv6（可用 `ipaddress.IPv6Address`）。
- modify 的 prompt 默认值来自 DB：若 DB 已是 `""`（空字符串），用户直接回车会保留空值；这对 endpoint 还好，但对 PublicKey 可能致命。

## 5. 安全与权限（Security & Permissions）

**做得好的点**

- DB 文件与含私钥的配置文件均尝试设置 `0600`（`db.py:Database.open`、`services._write_text(mode=0o600)`、`config.save_config`）。
- 调用 `wg`/`systemctl` 使用 list 形式参数，基本避免 shell 注入风险。

**需要关注的点**

- systemd unit 生成里将路径直接拼到 `ExecStartPre/ExecStart`：若用户通过参数把 `bird_roa_v6_conf_path` 指到带空格的路径，会导致 unit 解析失败。
  - 建议：对路径做约束（只允许无空格）或在 unit 里用更稳妥的 quoting 方案（但要避免引入 shell 注入）。
- ROA 下载失败时会写入 placeholder 文件（纯注释）。在当前 Bird 模板中，ROA 检查逻辑是 “非 ROA_VALID 直接 reject”，因此 **空表会导致所有路由被判定为 UNKNOWN 并拒绝**。
  - 这可能比“include 文件不存在导致 Bird 启动失败”更隐蔽。
  - 建议：明确策略：
    - 要么模板改成只拒绝 `ROA_INVALID`，对 `UNKNOWN` 放行；
    - 要么在 ROA 不可用时禁用该过滤（需要额外开关/模板分支）；
    - 或至少在 init 输出中强调“ROA 未就绪会导致路由导入为 0”。

## 6. 错误处理与健壮性（Robustness）

- `scan_local_configs` 的解析目前偏“强失败”：
  - `configparser.ConfigParser.read_string()` 在遇到坏格式可能抛异常；
  - `sanitize_name()` 在遇到异常 ifname（例如 `wg__`）会抛 `Dn42CtlError`，可能中断整个 scan。
  - 建议：scan 更适合“逐接口 best-effort”：对单个 ifname 的解析异常应降级为 `warnings/skip`，避免“一处坏文件拖垮全局”。
- `show_*` 的 live 命令调用未设置 timeout：理论上 `birdc/wg` 若阻塞会卡住 `dn42ctl show`。
  - 建议：增加短 timeout（例如 1~2s），并将超时作为 `CommandOutput(ok=False, error="timeout")`。

## 7. 可维护性与仓库卫生（Maintainability & Repo Hygiene）

- 根目录存在 `create_peer.py`、`setup_dn42_roa.sh` 与多份 `*_template`：
  - 它们与 `src/dn42ctl/*` 的实现可能**不一致**（例如旧脚本未体现 `RouteTable=off` / `peer-routes=false` 等硬约束）。
  - 建议：
    - 标记为 legacy（README/注释说明不再维护），或迁移到 `examples/`；
    - 或直接删除/归档，避免用户误用造成配置偏离 spec。
- `services.py` 文件体积较大（init + peer + show + del + scan + parsers 都在一起）。
  - 建议：按领域拆分（例如 `services/show.py`, `services/scan.py`, `services/roa.py`），但这属于中期重构。

## 8. 可测试性（Testability）

当前缺少测试。由于渲染与解析逻辑都已较纯，建议最低限度补：

- `render.py`：
  - NM 输出必须包含 `peer-routes=false`、`allowed-ips` 末尾分号；
  - networkd 输出必须包含 `RouteTable=off`；
  - endpoint 为空时必须省略 `Endpoint=` 行。
- `services.py` scan parsers：
  - `_parse_networkd_netdev/_parse_nmconnection/_parse_wgquick_conf` 对典型样例的字段抽取。
